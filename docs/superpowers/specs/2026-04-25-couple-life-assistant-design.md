# 开源自托管情侣生活助手设计规格

日期：2026-04-25

## 目标

构建一个开源自托管的微信小程序，用于同居情侣管理日常生活。项目默认每个部署实例只服务一对情侣，数据保存在部署者自己的设备或服务器中，优先支持部署到香橙派，并通过 Docker 和 CI/CD 方便后续快速迭代。

产品重点不是做公开 SaaS，而是提供一套可复制部署的私密情侣生活系统：天气、喝水提醒、快递共享、共同记账、外卖决策、AI 生活助手和可扩展 MCP 工具。

## 核心原则

- 开源自托管：不依赖中心服务器，不收集用户数据。
- 单实例一对情侣：权限模型简单，默认只有两名成员。
- 普通功能不依赖 AI：AI 出问题时，天气、记账、快递、提醒等核心功能仍可使用。
- AI 能力可选：在线大模型 API 优先实现，Ollama 作为可选适配预留。
- MCP 后端运行：小程序前端只调用后端 API，不直接接触模型密钥或 MCP 工具。
- 工程化优先：Docker 部署、环境变量配置、数据库迁移、健康检查、测试和 CI/CD 从第一版开始规划。

## V1 功能范围

### 首页：我们的一天

首页作为日常入口，聚合当天最重要的信息：

- 今日天气和简单生活建议。
- 今日待办。
- 待取快递。
- 喝水状态和提醒入口。
- 今晚吃什么推荐入口。
- 最近一笔共同支出。
- 纪念日或重要倒计时的轻提示。

首页下方保留时间线，用于展示当天发生过的生活事件，例如吃了什么、记了一笔账、取了快递、完成了待办、AI 新增了一条记忆。

### 吃什么系统：外卖优先

同居情侣的高频问题是“今天点什么外卖”。V1 以外卖决策为主，自己做饭作为后续扩展。

核心能力：

- AI 根据历史饮食、口味偏好、天气、预算、最近吃过内容生成 3 个推荐。
- 推荐分为“最快拍板”“她可能喜欢”“今天适合吃”。
- 用户仍纠结时进入转盘，由系统按偏好权重抽取店铺或菜品。
- 吃完后可以对 AI 说“今天吃了什么、多少钱、好不好吃”，AI 解析并请求用户确认后保存。
- 外卖支出可以一键进入共同记账。

V1 不接入美团、饿了么真实接口，不做自动下单。常点店铺、菜品、踩雷记录先通过手动录入和 AI 对话沉淀。

### AI 饮食记忆

AI 不能把记忆只放在模型供应商侧。系统需要把记忆保存到自托管数据库，并可检索、查看、编辑和删除。

V1 支持 3 类记忆：

- 吃过记录：日期、店铺、菜品、金额、评价、是否踩雷。
- 口味偏好：喜欢的菜系、忌口、不喜欢的食材、辣度、油腻接受度、甜咸偏好。
- 推荐反馈：喜欢、不想吃、下次少推荐、下次多推荐。

保存流程：

1. 用户用自然语言告诉 AI。
2. AI 解析为结构化候选记录。
3. 用户确认后写入数据库。
4. 系统生成或更新向量记忆索引。

### 共同记账

V1 做轻量共同账本：

- 新增支出：金额、分类、支付人、备注、日期。
- 分类优先覆盖外卖、买菜、日用品、房租、水电、交通、娱乐。
- 外卖记录可一键生成支出。
- 首页显示最近支出，统计页显示月度总额和分类占比。

V1 不做复杂 AA 结算和支付功能，后续再扩展。

### 快递共享

用于解决“帮我拿一下快递”：

- 发布快递：快递名、取件码、驿站位置、备注、发布人。
- 状态：待取、已取、已取消。
- 支持提醒对方拿快递。
- 完成后进入时间线。

V1 不接入真实物流 API。

### 喝水提醒

用于轻量关心，不做强制打卡工具：

- 每个人可记录喝水次数。
- 可提醒对方喝水。
- 可设置每日提醒上限，避免打扰。
- 首页显示双方当天状态。

微信订阅消息能力通过统一提醒系统接入；若用户未授权订阅消息，则退化为站内提醒。

### 待办和纪念日

V1 支持基础共同待办：

- 待办标题、负责人、截止时间、状态。
- 完成后进入时间线。

纪念日支持：

- 名称、日期、重复规则、提醒时间。
- 首页只做轻提示，不喧宾夺主。

## AI Agent 与 MCP

### Agent 定位

AI Agent 是“生活小管家”，负责把系统模块串起来，而不是替代基础功能。

V1 场景：

- “今天吃了麻辣烫，花了 45，她觉得不错但不要太辣。”
- “今晚吃什么？”
- “明天提醒我拿快递。”
- “今天有什么事？”
- “把今晚外卖记到账本里。”

### 模型适配

V1 优先实现 OpenAI-compatible 在线 API，配置项预留 Ollama：

- `LLM_PROVIDER`
- `LLM_BASE_URL`
- `LLM_API_KEY`
- `LLM_MODEL`
- `EMBEDDING_PROVIDER`
- `EMBEDDING_BASE_URL`
- `EMBEDDING_API_KEY`
- `EMBEDDING_MODEL`
- `OLLAMA_BASE_URL`

### MCP 设计

MCP 工具只在后端运行。前端请求后端，后端再由 Agent 调用工具。

V1 内置或预留这些工具接口：

- 天气工具：查询本地天气。
- 饮食记忆工具：读取和写入饮食记录、偏好、反馈。
- 记账工具：创建支出记录。
- 待办工具：创建和查询待办。
- 快递工具：创建和查询快递。

天气 MCP 作为示例工具实现，普通首页天气仍可直接由后端 API 提供并缓存。

## 数据与向量记忆

系统采用主数据库加向量索引的设计。

主数据库保存真实业务数据：

- 用户与情侣关系。
- 天气缓存。
- 支出记录。
- 快递记录。
- 喝水记录。
- 待办和纪念日。
- 外卖店铺、菜品、吃过记录、偏好、反馈。
- AI 对话摘要和确认后的记忆。

向量库只用于 AI 语义检索：

- 记忆文本。
- embedding。
- 记忆类型。
- 关联业务记录 ID。
- 创建时间、更新时间。

V1 默认数据库使用 SQLite，适合香橙派和单情侣实例。默认向量实现也放在 SQLite 中：`memory_embeddings` 表保存 embedding 向量、元数据和关联业务记录 ID，后端在数据量较小时做 Top-K 相似度检索。这样不需要额外向量数据库容器，香橙派部署更轻。

向量层必须通过接口封装，后续可以替换为 sqlite-vec、pgvector 或 Qdrant。Qdrant 作为可选 Docker Compose 扩展，不作为 V1 必需依赖。

## 技术架构

### 技术栈

V1 默认采用 TypeScript 全栈，减少前后端类型割裂：

- Monorepo：pnpm workspace。
- 小程序：原生微信小程序 + TypeScript。
- 后端：Node.js 22 + Fastify + TypeScript。
- 数据库：SQLite + Drizzle ORM。
- 任务调度：后端内置轻量定时任务。
- AI Provider：OpenAI-compatible API 优先，Ollama 适配预留。
- 测试：Vitest 覆盖核心业务逻辑和 Agent 工具。
- 容器：Docker 多阶段构建，支持 linux/arm64。

### 前端

微信小程序前端负责页面展示和交互：

- 首页。
- 吃什么推荐和转盘。
- AI 小管家聊天。
- 记账。
- 快递。
- 喝水。
- 待办。
- 设置和健康检查。

前端不保存敏感密钥，不直接调用 LLM 或 MCP。

### 后端

后端提供 REST API 和 Agent 服务：

- 用户绑定和会话。
- 业务模块 API。
- AI Agent 编排。
- MCP 工具调用。
- 数据库迁移。
- 任务调度和提醒。
- 健康检查。

后端实现应保持轻量，避免引入对香橙派不友好的重型运行时或必须额外维护的服务。除可选 Qdrant 外，V1 默认只需要后端容器和数据卷。

### 部署

目标部署环境是香橙派，默认使用 Docker Compose：

- `app-api`：后端服务。
- `app-db`：V1 可省略独立数据库容器，使用挂载卷中的 SQLite 文件。
- `app-data`：上传文件、数据库、备份目录。
- 可选 `qdrant` 或其他向量服务。

镜像需要支持 `linux/arm64`。CI/CD 构建时应考虑多架构镜像，至少覆盖 `linux/amd64` 和 `linux/arm64`。

### CI/CD

仓库应从第一版开始保留工程化流水线：

- 代码检查。
- 单元测试。
- 构建后端 Docker 镜像。
- 构建或校验小程序代码。
- 生成多架构镜像。
- 推送到容器镜像仓库。
- 香橙派通过 Watchtower、Webhook 或手动 `docker compose pull && docker compose up -d` 更新。

V1 可以先提供手动部署脚本，后续再接入完整自动发布。

## 隐私与安全

- 默认不启用公网公开注册。
- 单实例只允许一对情侣绑定。
- 邀请码用于绑定另一半。
- AI 记忆写入必须用户确认。
- 设置页提供记忆查看、编辑、删除、关闭。
- 敏感配置只放在服务端 `.env`。
- 备份文件不应提交到 Git。
- 后端 API 需要基础鉴权，避免局域网内任意访问。

## V1 非目标

- 不做多情侣 SaaS 平台。
- 不做社区广场。
- 不接入美团、饿了么真实接口。
- 不做自动下单。
- 不做支付、会员、公开内容审核。
- 不让 AI 未确认就保存敏感记忆。
- 不把所有业务数据只存向量库。

## 验收标准

- 能通过 Docker Compose 在香橙派或本地 ARM64 Linux 上启动后端。
- 微信小程序能配置后端地址并完成情侣绑定。
- 首页能展示天气、待办、快递、喝水和吃什么入口。
- 用户能用 AI 记录一条饮食记忆，并在确认后保存。
- 吃什么系统能基于历史记录给出 3 个推荐，并提供转盘拍板。
- 外卖记录能一键进入记账。
- 快递和喝水功能可完成基本闭环。
- 设置页能查看服务健康状态和 AI 记忆。
- 仓库具备基础测试、Dockerfile、Compose 文件和部署文档。
